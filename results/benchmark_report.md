# Benchmark Report

## Core Assumptions and PM Cues

1. This report blends speed and code validity. Overall Score is weighted 50% efficiency and 50% compilation pass-rate.
2. Compilation Pass % indicates syntax-valid Python output only; it does not guarantee classroom-ready behavior quality.
3. Overfit Pass % is computed from template-like or explicitly overfit-tagged cases and reflects performance on narrow, constrained prompts.
4. Runs with memory-threshold aborts are excluded from ranking averages and appear in Detailed Runs with aborted status.
5. Compare models first by pass metrics for reliability, then by TTFT/Tokens/s/RAM for deployment fit on this hardware.

## Abbreviations and Settings Glossary

- TTFT: Time To First Token. Lower is better for perceived responsiveness.
- Tokens/s (TPS): Tokens generated per second after generation starts. Higher is better for throughput.
- Peak RAM (GiB): Highest observed RAM usage during the run. Lower is better for 8GB-class systems.
- Compilation Pass %: Percentage of runs where generated Python passed syntax compilation.
- Passes: Raw pass count shown as passed/total runs in the ranking table.
- Overfit Pass %: Pass percentage on template-like or explicitly overfit-tagged cases only.
- Efficiency Score: Normalized blend of TTFT, TPS, and Peak RAM.
- Overall Score: 50% Efficiency Score + 50% Compilation Pass %.
- Temperature: Sampling randomness. Lower values are more deterministic.
- num_predict: Maximum output tokens allowed for a generation.
- num_ctx: Model context window used for prompt + generation.
- top_k/top_p: Token sampling filters; they constrain token choice breadth.
- repeat_penalty: Reduces repetition by penalizing previously used tokens.
- Warm-up: Optional pre-run model touch used to reduce cold-start latency skew.
